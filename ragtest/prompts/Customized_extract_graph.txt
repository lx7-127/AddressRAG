-Goal-
Given an AddressRAG semantic text chunk and a list of allowed entity types, extract the toponym-address entities and the explicit relationships among them. The result will be parsed by the GraphRAG entity-and-relationship extraction pipeline.

-Allowed Entity Types-
Use only entity types included in [{entity_types}]. Never invent an entity type outside this list.

The recommended AddressRAG vocabulary is:
- TOPONYM: a named POI, landmark, natural feature, building, facility, or other named geographic entity.
- PROVINCE: a province-level administrative area.
- CITY: a city-level administrative area.
- DISTRICT: a district- or county-level administrative area.
- TOWNSHIP_OR_SUBDISTRICT: a township, town, or subdistrict-level administrative area.
- ROAD: a named road, street, avenue, or lane.
- ADDRESS_ELEMENT: a house number, building, development zone, intersection, internal POI location, or other fine-grained address component that is not better represented by another allowed type.
- COORDINATE: one complete WGS84 longitude-latitude pair. Treat the pair as one entity, not as separate longitude and latitude entities.
- POI_CATEGORY: a category in the POI classification hierarchy.

If a recommended type is not present in [{entity_types}], use the closest semantically valid allowed type. If no valid type is available, retain the information in the description of a related entity instead of creating an entity with an unsupported type. For example, if COORDINATE is not allowed, retain the WGS84 values in the TOPONYM description.

-Extraction Rules-
1. Extract entities.
For each entity, provide:
- entity_name: a stable canonical name. Preserve the source-language toponym or address string; trim surrounding whitespace; uppercase Latin-script names for consistent linking. Do not translate or paraphrase entity names.
- entity_type: exactly one type from [{entity_types}].
- entity_description: a concise English description containing only facts stated in the input. For a TOPONYM, retain available administrative affiliation, detailed address, WGS84 coordinates, and POI classification. For an administrative area, state its level and explicit parent. For a POI category, state its explicit position in the category hierarchy.

Format each entity as:
("entity"{tuple_delimiter}<entity_name>{tuple_delimiter}<entity_type>{tuple_delimiter}<entity_description>)

2. Extract explicit relationships only between entities emitted in Step 1.
Use the following direction and wording when supported by the input:
- ADMINISTRATIVELY_BELONGS_TO: TOPONYM to its most specific stated administrative area, or a lower administrative area to its immediate parent.
- LOCATED_AT: TOPONYM to a detailed ADDRESS_ELEMENT.
- LOCATED_ON: TOPONYM or ADDRESS_ELEMENT to a ROAD.
- HAS_COORDINATE: TOPONYM to its COORDINATE.
- HAS_CATEGORY: TOPONYM to its most specific POI_CATEGORY.
- SUBCATEGORY_OF: a more specific POI_CATEGORY to its immediate parent category.
- NEAR: TOPONYM to an explicitly named nearby TOPONYM or landmark.
- WITHIN: an internal POI or ADDRESS_ELEMENT to an explicitly named containing TOPONYM.

Begin each relationship_description with the applicable uppercase relationship label followed by a colon, then explain the evidence in English. The relationship label remains part of the description because the GraphRAG tuple schema does not contain a separate relationship-type field.

Assign relationship_strength on a 1-10 scale:
- 10: a direct record-level fact, including explicit administrative affiliation, detailed address, or coordinate assignment.
- 9: an explicit administrative or category hierarchy relation.
- 8: an explicit containment, road, intersection, or proximity relation.
- 6-7: another unambiguous relation directly supported by the text.
- 1-5: too weak for this task; do not emit the relationship.

Format each relationship as:
("relationship"{tuple_delimiter}<source_entity>{tuple_delimiter}<target_entity>{tuple_delimiter}<relationship_description>{tuple_delimiter}<relationship_strength>)

3. Apply the following quality constraints:
- Use only evidence in the input text. Do not infer missing administrative levels, addresses, coordinates, categories, aliases, or spatial relations.
- Treat null, nan, unknown, unavailable, and redacted values as missing. Do not create entities or relationships from them.
- Do not create a relationship merely because two entities occur in the same text chunk.
- Do not convert generic boilerplate, such as a statement that a POI forms part of a local service topology, into an edge unless a named target and a specific relation are stated.
- Emit each entity once per input chunk. Merge multiple supported attributes into one description.
- Emit one relationship tuple per distinct source-target pair. If several facts connect the same pair, combine them in one concise description.
- Source and target names must exactly match entity_name values emitted in Step 1.
- Do not place {tuple_delimiter}, {record_delimiter}, or {completion_delimiter} inside a field value.
- Do not output empty records, duplicate delimiters, Markdown, headings, explanations, or any text outside the required records.

4. Return the output in English as one sequence of entity and relationship records. Use {record_delimiter} exactly once between adjacent records. After the final record, output {completion_delimiter} without an additional record delimiter.
